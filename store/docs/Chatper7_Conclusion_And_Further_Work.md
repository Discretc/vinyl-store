# Chapter 7 Conclusion and Further Work

## 7.1 Summary

This project set out to develop a multi-vendor online shopping platform for vinyl records, implementing three basic requirement blocks (A, B, F) and three further requirement blocks (T, U, X) from the system specification. The final delivered system is a Django-based web application that supports end-to-end customer shopping and vendor operations within a single integrated platform.

Block A required basic online shopping functions. The system satisfies this through customer registration and authentication, a searchable and filterable product catalogue presented in a responsive grid, product detail pages with additional attributes beyond the listing view, a server-side shopping cart with quantity management and item removal, checkout with order creation, and an order history page that lists past purchases in reverse chronological order with full detail views. The system also supports anonymous browsing before login as specified, and cart data persists across sessions on the server side. The product list supports pagination on desktop and infinite scroll on mobile, meeting the requirement for suitable navigation controls on different device types. Beyond the minimum specification, the system also implements selective checkout where customers choose specific cart items via checkboxes, and unselected items remain in the cart.

Block B required multiple product photos and basic order processing. The system supports uploading multiple images per product, designating a primary image, and displaying a product image gallery on the detail page. Order processing is implemented through a five-stage status model — Processing, Holding, Shipping, Completed, and Cancelled — with a full status history recorded per order item rather than a single overwritable field. Customers can filter orders by status, and status-change dates are stored and displayed in a timeline. The customer-facing order page supports cancellation of items in eligible statuses and refund requests for shipped or completed items, while vendors can update statuses and respond to refund requests from their administration portal.

Block F required multiple vendors and shops. The system allows vendors to register accounts, create shops with descriptive information and gallery photos, and manage their own separate product catalogues. Customers can visit public shop pages to browse within a single vendor's inventory and search within a shop. The shopping cart accepts products from different shops, and the selective checkout mechanism allows customers to choose which items to purchase. Each vendor's administration portal is scoped to their own store, so vendors can only manage their own products, images, promotions, and order items. This separation is enforced in every vendor-facing view through ownership verification.

Block T required user-generated content. The system implements this through a review and rating system where customers who have purchased a product can submit a star rating, an optional written comment, and an optional review photo. Review photos are resized on the server before storage. Customers can also delete their own reviews. Ratings and reviews are displayed on the product detail page and contribute to per-product average ratings shown across the catalogue.

Block U required a wishlist and promotional pricing strategy. Customers can add and remove products from a wishlist, and the wishlist page displays current pricing alongside the price recorded when the item was added. Vendors can create percentage-based promotions with defined start and end dates, toggle promotions between active and inactive states, and the system automatically expires promotions that have passed their end date. Discounted prices are reflected consistently across the home page, product listing, product detail, cart, checkout, and wishlist. When a promotion activates on a wishlisted product, the system generates a notification to the affected customers. The wishlist page also sorts promoted items to the top by discount rate, and customer search queries are logged and surfaced in the vendor dashboard to support demand anticipation.

Block X required user experience design. The system addresses this through responsive layouts at two breakpoints, a shared base layout that provides consistent navigation and visual patterns, role-specific interface separation for customers and vendors, colour-coded status labels, notification badges and dropdowns for both user types, toast-style feedback messages, and ARIA attributes on interactive controls. The home page is structured with separate promotional and featured product sections to support different browsing intentions.

## 7.2 Limitations and Further Work

Despite satisfying the selected requirement blocks, the system has several limitations that represent opportunities for future development.

The most significant technical limitation is the absence of an automated test suite. Verification was performed manually throughout development, as described in Chapter 6. While this was sufficient to confirm correctness during the project, it leaves the system vulnerable to regressions if features are modified in the future. The priority further work in this area would be to convert the existing manual test scenarios into automated unit and integration tests, particularly around checkout pricing, stock management, order lifecycle transitions, permission checks, and notification generation.

The system does not include a real payment processing integration. Checkout creates an order and reduces stock, but no payment gateway or payment confirmation step is involved. In a production extension, a payment service such as Stripe or PayPal could be integrated into the checkout flow, with order creation conditional on successful payment. This would also require handling payment failures and partial-payment edge cases.

The notification system is currently limited to in-application notifications. There is no email or push notification delivery. A useful extension would be to send email notifications for high-priority events such as order confirmations, shipment updates, and wishlist promotions. For mobile users, push notifications through a service worker or a dedicated mobile application would further improve engagement, as suggested by the Block X specification.

The security measures described in Chapter 6 are appropriate for a coursework application but would need hardening for production deployment. Specific improvements include disabling debug mode, restricting allowed hosts, enforcing HTTPS, adding rate limiting on login and API endpoints, improving file upload validation beyond image resizing, and replacing the `innerHTML`-based notification rendering with safer DOM construction methods.

Product discovery could be strengthened in several ways. The current search is based on substring matching against product names and descriptions. A more capable system could implement full-text search with relevance ranking, product recommendations based on browsing and purchase history (Block S), or category and tag-based filtering (Block C). The click history data already collected by the system provides a foundation for building recommendation features in future iterations.

The vendor analytics on the dashboard are currently basic summaries of wishlist counts, view counts, and search terms. Extending this toward the reporting and analytics capabilities described in Block V would give vendors more actionable business intelligence, such as sales trends over time, conversion rates from views to purchases, and comparative product performance.

Finally, accessibility was addressed through responsive design, focus styling, and ARIA attributes, but the system was not formally audited against WCAG guidelines. A structured accessibility review with assistive technology testing would identify remaining gaps, particularly around keyboard navigation completeness, colour contrast ratios, alternative text for user-uploaded images, and reduced-motion support. Internationalisation was also not implemented, so the interface is English-only.

In summary, the delivered system meets the requirements of the six selected blocks and provides a working multi-vendor shopping platform with integrated customer engagement features. The limitations identified above are realistic extensions rather than gaps in the core specification, and the existing architecture provides a reasonable foundation for addressing them incrementally.

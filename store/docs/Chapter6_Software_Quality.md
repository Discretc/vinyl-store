## 6.1 Software Verification

The group used a mainly manual, requirement-based verification approach to check whether the implemented system behaved correctly against the selected specification blocks. In this project, verification was carried out by the members responsible for each functional area during development, followed by cross-checking by other group members when the integrated workflows were demonstrated. Since the codebase does not include a developed automated test suite in `store/tests.py`, the dominant testing method for this project was manual testing supported by debugging, repeated scenario execution, and direct inspection of database results and browser behaviour.

The testing approach combined four practical layers. First, debugging was used during implementation to trace logic errors, inspect submitted form data, verify session values, and confirm database state transitions after major actions such as checkout, order-status updates, review submission, and refund handling. Second, view-level and model-level validation were checked manually as functional tests, for example by verifying stock reductions after checkout, stock restoration after cancellation or approved refund, prevention of duplicate wishlist entries, and correct calculation of discounted prices. Third, integration testing was performed by executing complete end-to-end workflows that crossed multiple pages and models, such as customer registration to product browsing to selective checkout to vendor order processing. Fourth, acceptance-style testing was carried out by comparing implemented features with the selected requirement blocks to ensure that the final system actually delivered the expected outcomes for Blocks A, B, F, T, U, and X.

The test cases were designed around the non-trivial user journeys and business rules of the system rather than around isolated page loading only. For example, checkout testing did not stop at verifying that an order page opened correctly. It included confirming that only the selected cart items were converted into order items, that unselected items remained in the cart, that paid prices reflected active promotions, that stock quantities were reduced correctly, and that vendor notifications were generated for the affected stores. Similarly, order-processing tests checked that vendors could only manage order items belonging to their own store, that status histories were appended instead of overwritten, and that customers could see the updated results from their own account.

Requirement-based tests were especially important because the selected blocks contain several features that only become meaningful when combined. Multiple product photos were tested not only by uploading images but also by confirming that the correct primary image was shown in product cards and detail pages. Wishlist and promotional pricing were tested together by checking whether current discounts propagated consistently across the home page, product listing, product detail page, cart, checkout, and wishlist. User-generated content was tested by verifying that only customers with a purchase record could post reviews, that uploaded review photos were resized correctly, and that deleting a review also removed the stored photo file.

Major functions were verified with both valid and invalid data. Registration and profile forms were tested with missing required fields, mismatched passwords, and duplicate email addresses. Cart operations were tested with invalid quantities such as zero, negative, and greater-than-stock values. Promotion management was tested with invalid discount percentages and end dates earlier than start dates. Refund requests were tested against ineligible statuses to confirm that customers could only request refunds for shipped or completed items. These checks were supported by both Django model validators, such as minimum and maximum value constraints, and view-level validation logic that returned either error messages or JSON error responses depending on the interaction style.

Sample data was used to support repeatable verification without making the test process unnecessarily repetitive. The project includes a seeding command that creates multiple customers, vendors, stores, products, promotions, and reviews. This made it possible to test realistic multi-vendor scenarios, such as a single order containing items from different stores, wishlist behaviour across several products, and vendor analytics based on customer search and browsing activity. Additional edge-case data was then entered manually when needed, such as products with zero stock, products without images, expired promotions, or customers with no prior purchases. This combination of seeded baseline data and targeted edge-case data allowed the group to test robustness more efficiently than re-entering the same setup for every scenario.

From a software-quality perspective, the main strength of the verification process is that it focused on the correctness of the implemented workflows and business rules. The main limitation is that automated regression coverage is still weak. Since the project currently relies on manual verification, future changes would be safer if the current requirement-based scenarios were converted into automated unit and integration tests, especially for pricing, permissions, notifications, and order lifecycle behaviour.

## 6.2 Security

Security in the system was addressed primarily through framework-supported protection, input validation, and ownership checks on sensitive actions. Because the system handles account data, order information, uploaded media, and vendor-specific operations, the main risks considered were unauthorized access, malicious form submission, cross-site scripting, cross-site request forgery, and incorrect manipulation of order or pricing data.

For data security, customer and vendor passwords are not stored in plain text. They are hashed before storage using Django's password hashing utilities, and login checks compare submitted passwords against the stored hashed value. The system also uses server-side sessions, so authentication state is maintained on the server rather than trusting browser-side data alone. In addition, sensitive operations are guarded by role-specific session checks. Customer-only functions, such as cart, wishlist, order history, review submission, cancellation, and refund request, require a valid customer session. Vendor-only functions, such as dashboard access, product editing, promotion creation, order-status updates, and refund response, require a valid vendor session.

Beyond login checks, the system applies ownership verification to reduce unauthorized data access. For example, vendors can update only products, images, promotions, refund requests, and order items that belong to their own store. Customers can access only their own orders, cancel only their own eligible order items, and delete only their own reviews. Notification read actions also verify that the notification belongs to the logged-in recipient. These controls are important because they protect against insecure direct object reference problems where a user might otherwise try to manipulate another user's record by changing an identifier in the URL.

To mitigate malicious or malformed input, the system uses a combination of Django model constraints and explicit validation in views. Numeric fields such as prices, quantities, ratings, and discount rates use validators to enforce sensible ranges. Form handlers also check required fields, password confirmation, stock limits, shipping address presence, and promotion date order. Invalid data is rejected before it can alter the system state. Uploaded review photos are processed and resized on the server before storage, which helps standardize uploaded content and reduces the risk associated with accepting arbitrary image files directly into the interface.

Cross-site scripting risk is reduced mainly through Django's template system, which escapes output by default. User-generated content such as review comments, notification messages, store descriptions, and search queries is rendered through templates rather than being inserted as raw HTML. The codebase also does not rely on the `safe` filter to bypass output escaping for these fields. From the external perspective of the system, this means that user-supplied text is displayed as text rather than being executed as script in the browser.

One area where this protection is weaker is the notification dropdown, which renders notification items in the browser using JavaScript `innerHTML` with data returned from a JSON endpoint. Although the notification content originates from server-generated messages rather than from direct user input, some of those messages include user-controlled values such as product names and customer names. If a malicious user were to set a product name containing script markup, the notification dropdown could render it without escaping. This is a minor residual risk because the attack surface is limited to logged-in notification recipients, but a more robust approach would be to use DOM text-content methods or a client-side escaping function when building notification HTML dynamically.

Cross-site request forgery protection is provided through Django's `CsrfViewMiddleware` together with CSRF tokens in standard forms and AJAX requests. POST-based operations such as cart updates, wishlist changes, review submission, promotion management, status updates, and notification state changes send CSRF tokens either through hidden form fields or request headers. This ensures that state-changing actions cannot be triggered legitimately from an unrelated external site without the correct token. In addition, the middleware stack includes `XFrameOptionsMiddleware`, which sets a header to prevent the application pages from being embedded in frames on external sites. This reduces the risk of clickjacking attacks where a malicious site could overlay invisible controls on top of the application to trick users into performing unintended actions.

State-changing endpoints are also restricted to the correct HTTP method using Django's `@require_POST` decorator. This is applied consistently across approximately twenty view functions, including cart updates, checkout, review submission, wishlist toggling, order cancellation, refund requests, status updates, promotion management, and notification actions. By rejecting non-POST requests at the decorator level, the system prevents accidental or malicious triggering of write operations through GET links, browser pre-fetching, or bookmarked URLs.

The project did not include a formal stress test or denial-of-service test campaign, so no measured performance claims should be made about resistance to large-scale attack traffic. However, some practical design choices still reduce accidental misuse, such as requiring explicit POST requests for state-changing operations and validating inputs before database writes occur. At the same time, there are still clear limitations if the system were to be deployed publicly in production. In the current development configuration, `DEBUG` is enabled and `ALLOWED_HOSTS` is set broadly, which is acceptable for coursework development but not for an internet-facing deployment. A production version would need tightened host configuration, debug disabled, HTTPS deployment, stronger upload restrictions, and more structured logging and monitoring.

Overall, the security approach is appropriate for a coursework web application because it addresses the main practical threats at the application level: password protection, request forgery prevention, escaped output, validation of user input, and role-based authorization. The main remaining improvements relate to production hardening rather than to the absence of basic security measures.

## 6.3 Reflection on User Experience

The user experience of the system was designed with the expectation that the platform could be used by customers and vendors with different levels of digital confidence. A vinyl record marketplace may attract both younger users who are comfortable with online shopping patterns and older users who may value clearer navigation, stable layouts, and predictable transaction feedback. For that reason, the system emphasizes familiar e-commerce structures such as visible search and filtering, product cards, cart summaries, order history pages, and clear action buttons rather than unusual interaction patterns.

For users from different cultural and demographic backgrounds, the interface attempts to remain understandable by keeping the navigation structure simple and by using conventional shopping language such as Browse, Wishlist, Orders, Dashboard, and Logout. The layout is also responsive so that customers browsing casually on mobile devices and vendors managing products on larger screens can both use the same application effectively. The separation between customer and vendor workflows also improves clarity because users are not overloaded with actions that are irrelevant to their role.

Several design decisions support accessibility, although the system should be understood as partially accessible rather than fully accessibility-audited. The interface uses readable page structures, consistent navigation, visible feedback messages, and focus styling on form controls so that keyboard-based interaction is easier to follow. The mobile navigation and notification controls include ARIA attributes such as `aria-label` and `aria-expanded`, which improves interpretation by assistive technologies. Buttons and form fields are visually distinct, and the layout adapts at tablet and mobile breakpoints to reduce horizontal overflow and cramped interaction targets.

The design also considered users with motor or visual limitations to a practical extent. Important actions such as add-to-cart, filter submission, checkout, status updates, and promotion management are presented as explicit buttons or forms rather than relying only on gesture-like interactions. Notification badges, order-status labels, and grouped summaries help users identify important changes without re-reading large sections of text. Form feedback through alerts and toast-style messages also reduces ambiguity after an action has been performed.

At the same time, there are limitations that should be acknowledged honestly. The system does not claim full compliance with accessibility standards such as WCAG 2.1, and no formal assistive-technology evaluation was conducted. Some improvements would still be needed for a stronger inclusive design outcome, such as more comprehensive keyboard-only testing, explicit reduced-motion support, stronger colour-contrast auditing, more descriptive alternative text strategy for uploaded images, and screen-reader testing across the most important workflows.

The testing process for user experience in this project was primarily based on internal walkthroughs rather than a formal study involving a large external user sample from many backgrounds. Therefore, it would be inaccurate to claim extensive user research across age groups, cultures, or disabilities. A more defensible conclusion is that the system was designed with inclusive intentions and tested for practical usability across desktop and mobile contexts, but broader user validation remains future work. If the project were extended, the next step would be to run task-based usability sessions with participants of different experience levels and accessibility needs, then refine the interface based on observed difficulties.

## 6.4 Other Quality Attributes

Besides functional correctness, several other software quality attributes are important for this system because it is an interactive multi-vendor commerce platform rather than a simple static website. The most important additional qualities are maintainability, reliability, and performance.

### 6.4.1 Maintainability

Maintainability is important because the system contains many connected workflows, including shopping, reviews, wishlist management, promotion handling, notifications, and vendor operations. A change in one area can easily affect other areas, especially pricing and order lifecycle logic. The required level of maintainability for this project is moderate to high: the code should be understandable enough that future group members can safely modify features without breaking existing behaviour.

Several current design choices already support maintainability. The project uses Django models to structure domain entities clearly, URL routing to separate features, templates to isolate presentation, and helper logic such as promotion expiry and review-photo resizing to avoid repeating the same behaviour everywhere. However, there is still room for improvement because a large amount of business logic is concentrated in `views.py`. To ensure a stronger maintainability level in future, the project should gradually extract repeated logic into service functions or utility modules, document important workflows more explicitly, and add automated regression tests around the most sensitive business rules.

### 6.4.2 Reliability

Reliability is critical because customers expect accurate totals, correct order histories, and dependable status updates, while vendors expect their catalogue and order operations to behave consistently. The required level of reliability is high for core business operations, especially checkout, stock adjustment, promotion pricing, review eligibility, and refund handling. Errors in these areas would undermine trust in the system quickly.

The project supports reliability through validation, status-history modelling, and ownership checks. For example, order-item status changes are stored as separate records rather than overwriting history, and pricing logic preserves the paid price at the time of order creation. To ensure a stronger reliability level over time, future work should add automated tests for edge cases, use database transactions around critical multi-step operations such as checkout and refund approval, and expand error logging so unexpected failures can be diagnosed more quickly.

### 6.4.3 Performance

Performance matters because product browsing, cart calculation, notification loading, and vendor dashboard summaries are all user-facing interactions where delay is noticeable. The required level for this coursework project is responsive everyday performance under normal class-demonstration usage rather than enterprise-scale throughput. In practice, pages should load smoothly with seeded sample data, and interactive operations should complete quickly enough not to disrupt the shopping flow.

The system already includes some useful performance-oriented choices, such as query optimization through `select_related`, `prefetch_related`, pagination on desktop product browsing, and infinite scrolling on mobile to avoid loading an excessive number of product cards at once. Even so, performance quality could be improved further if the system grew in size. Possible measures include indexing frequently searched fields, caching high-traffic queries, limiting notification payload sizes more aggressively, and profiling dashboard queries that aggregate search, view, and wishlist data.

Overall, these non-functional qualities matter because they determine whether the system remains dependable and extensible after the initial implementation. A system that is functionally correct but difficult to maintain, unreliable under change, or slow during common operations would not provide strong overall software quality. For this reason, the project should be judged not only by whether features exist, but also by whether the code and architecture can support their continued correct use.
